# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Subscription Invoice Monitoring Agent (SIMA) — an AI agent that connects to a single Gmail
account, discovers subscription invoice emails from a config-driven vendor list, extracts
structured invoice data (deterministically where possible, via Claude where it genuinely requires
understanding), and persists invoices/attachments/processing history. Deployed as an Express HTTP
service on GCP Cloud Run, triggered daily by Cloud Scheduler.

This repo follows **Spec-Driven Development** via the spec-kit workflow (`.specify/`, `specs/`).
The active feature is `specs/001-gmail-invoice-ingestion/` (spec.md, plan.md, tasks.md,
data-model.md, contracts/http-api.md). Read `tasks.md` before doing implementation work in this
feature — it defines the exact task breakdown, file targets, and checkbox state (`[x]` done,
`[ ]` pending); update checkboxes as tasks complete. `.specify/memory/constitution.md` is the
project's binding engineering charter (13 principles) — notably: **Deterministic Before AI** (Claude
is scoped to invoice/vendor/metadata understanding only; auth, scheduling, storage, retries must
stay deterministic), **Idempotent Processing** (`SourceEmail.gmailMessageId` is the dedup key —
never let reprocessing create duplicate `Invoice` rows), and **Complete Auditability** (every
invoice must trace back to a `ProcessingHistoryEntry` and its source email).

## Commands

```bash
pnpm install                # install deps
pnpm run dev                # run server with hot reload (tsx watch)
pnpm run build               # tsc -> dist/
pnpm run start                # run compiled server (dist/index.js)
pnpm run lint / lint:fix     # eslint
pnpm run format / format:check  # prettier
pnpm run typecheck            # tsc --noEmit
pnpm run test                # vitest run (single pass)
pnpm run test:watch           # vitest watch
```

Run a single test file or test name with vitest directly, e.g.:

```bash
pnpm vitest run src/storage/attachmentStore.test.ts
pnpm vitest run -t "name of test"
```

Prisma:

```bash
pnpm prisma migrate dev       # create/apply a migration from schema.prisma changes
pnpm prisma migrate deploy    # apply existing migrations (CI/test DB setup)
pnpm prisma migrate reset     # wipe and reapply (test DB only)
```

The Prisma client is generated into `src/generated/prisma/` (not hand-edited; regenerated by
`prisma generate`, which `migrate dev`/`deploy` trigger automatically).

## Testing

`pnpm run test` runs Vitest, and it is **not just unit tests** — integration tests hit a real
PostgreSQL database via Prisma (constitution Principle X prohibits DB mocking) and the Gmail client
is tested against a faked `gmail_v1.Gmail` object, not a live account. Tests are co-located as
`*.test.ts` next to their source file (no top-level `tests/` dir).

`src/config/env.ts` runs `import 'dotenv/config'` at import time, so `DATABASE_URL` and every other
env var for `pnpm run test` come from the same `.env` used for `pnpm run dev` — there's no separate
test-env loading path. Before running tests that touch Prisma:

1. Point `.env`'s `DATABASE_URL` at a disposable Postgres instance/schema dedicated to tests (never
   dev/prod data) — see README.md for a `docker run postgres:16` one-liner.
2. `pnpm prisma migrate deploy` against it.
3. Re-run step 2 after pulling schema changes; `pnpm prisma migrate reset` to wipe it clean.

Tests that don't need a DB (e.g. `attachmentStore.test.ts`) mock `../config/env.js` directly with
`vi.mock` rather than depending on `.env`.

## Architecture

```
src/
  agent/
    invoiceMonitor.ts       # ingestion run orchestrator (entry point: runInvoiceCheck)
    gmail/client.ts         # OAuth2 Gmail API client (messages.list/get, attachments.get)
    gmail/discovery.ts      # vendor-driven query building + candidate discovery
    extraction/             # pdfExtractor, csvExtractor, aiExtractor (Claude-based)
    vendors/vendorConfig.ts # config-driven vendor identification (sender/subject patterns)
  storage/
    attachmentStore.ts      # AttachmentStore interface (save/retrieve/delete/getMetadata) — contract only
    gcsAttachmentStore.ts   # Cloud Storage implementation (used in all deployed envs)
    localAttachmentStore.ts # local-disk implementation (dev/test convenience only)
    prisma.ts               # Prisma client singleton (global-cached outside production)
  config/env.ts             # zod-validated env schema — the only way config enters the app
  lib/logger.ts             # pino structured logger
  server.ts                 # Express app + routes
  index.ts                  # process entry point (startServer())
  generated/prisma/         # generated Prisma client — do not hand-edit
prisma/schema.prisma        # Vendor, SourceEmail, Invoice, Attachment, ProcessingHistoryEntry
```

Data flow: Gmail discovery (vendor sender/subject rules) → attachment download → text extraction
(PDF/CSV deterministic parsers, falling back to Claude for genuine understanding) → structured
extraction validated with Zod → persisted as `SourceEmail` (upserted by `gmailMessageId`) +
`Invoice` + `Attachment` rows, with a `ProcessingHistoryEntry` written for every evaluated email
regardless of outcome (`PROCESSED` / `FAILED` / `SKIPPED_NOT_INVOICE` / `RETRYING`).

Key invariants enforced by the schema/design (don't casually change):

- `SourceEmail.gmailMessageId` is `@unique` — the idempotency key for the whole pipeline.
- `Invoice.sourceEmailId` is `@unique` — one invoice per source email.
- Every external input (Gmail API responses, Claude extraction output, env vars, HTTP requests)
  is parsed through a Zod schema before it enters business logic — see `GmailMessageSchema` /
  `GmailAttachmentSchema` in `src/agent/gmail/client.ts` and `envSchema` in `src/config/env.ts` for
  the pattern to follow for new external inputs.
- A failure on one email must never abort the rest of a run (per-email try/catch isolation).

HTTP surface: `GET /healthz` (liveness), `POST /tasks/check-invoices` (existing trigger — being
extended/replaced by `POST /tasks/ingest-invoices` plus `GET /invoices`, `GET /invoices/:id`,
`GET /processing-history` per `specs/001-gmail-invoice-ingestion/contracts/http-api.md`).

## Conventions

- Strict TypeScript throughout (`strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`
  all on) — respect these rather than widening types or reaching for `any` (ESLint disables
  `no-explicit-any` only inside `*.test.ts`).
- `@typescript-eslint/consistent-type-imports` is enforced — use `import type` for type-only
  imports.
- NodeNext ESM: relative imports must include the `.js` extension even though source is `.ts`.
- CI (`.github/workflows/ci.yml`) runs format:check → lint → typecheck → test → build, in that
  order, on every push/PR to `main`; lint-staged (husky pre-commit) runs eslint --fix/prettier on
  staged files. Match this order when checking your own work locally.
- Deploy (`.github/workflows/deploy.yml`) runs only after CI succeeds on `main`, building/pushing
  the Docker image to Artifact Registry and deploying to Cloud Run via Workload Identity
  Federation — there is no manual deploy step to replicate locally.

## Task Execution Guidelines

When implementing work from `specs/<feature>/tasks.md`, follow these rules.

### 1. Read Before Coding

Always review, in order:

1. `.specify/memory/constitution.md`
2. `spec.md`
3. `plan.md`
4. `research.md`
5. `data-model.md`
6. `contracts/`
7. `tasks.md`

If conflicts exist, stop and explain them before making changes.

---

### 2. Scope Control

Implement **only** the requested task(s).

Do not implement future tasks.

Do not modify unrelated files.

Do not introduce speculative abstractions.

Follow the project's YAGNI principle.

---

### 3. Dependency Awareness

Before implementation:

- Verify prerequisite tasks are complete.
- Respect task dependencies defined in `tasks.md`.
- If a dependency is incomplete, explain why implementation cannot continue.

Never bypass task ordering by implementing hidden prerequisites.

---

### 4. Validation

After every implementation:

- Run TypeScript type checking.
- Run ESLint.
- Run affected Vitest tests.
- Fix failures before considering the task complete.

Never leave the repository in a failing state.

---

### 5. Completion

When a task is finished:

- Update the task checkbox in `tasks.md`.
- Summarize:
  - Files modified
  - Tests added or updated
  - Validation performed
  - Acceptance criteria satisfied
  - Remaining limitations

Stop after completing the requested task(s).

Wait for the next instruction instead of continuing automatically.

---

### 6. Implementation Philosophy

Prefer small, reviewable changes.

Avoid combining multiple unrelated tasks into one implementation.

Each completed task should leave the repository in a deployable, testable state.

Favor readability, maintainability, and deterministic behavior over clever implementations.
